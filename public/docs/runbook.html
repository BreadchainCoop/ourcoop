The idea in one picture

Custom modules, canonical wiring

O.U.R.COOP is built from custom implementations of the standard crowdstake interfaces. You deploy those modules once, then the same creation wizard that builds a normal instance wires a distribution manager + cycle module around them.

Deploy the 5 COVA modules → Paste addresses into the wizard → Wizard deploys DM + cycle & wires all of it → Finish 3 follow-ups → Seed & use
The wizard's “Custom modules (advanced)” section has one field per interface slot — registry, token, cycle, voting module, distribution strategy, and a voting-power-strategies list. Empty fields deploy the canonical module; filled fields use yours.

Before you start

Prerequisites

1

Foundry

For the cast commands (and to re-deploy modules if you ever need to).

2

A funded Gnosis key

The coordinator — it owns the COVA modules. You've been using 0x6636…D9F0. Keep a little xDAI in it.

3

A browser wallet

MetaMask/Rabbit on Gnosis, connected as the coordinator, to drive the wizard.

4

Some xDAI to stake

A few xDAI to wrap → deposit as the cooperative's principal (backs the cUSD you'll distribute).

Everything below targets Gnosis mainnet. It's cheap (sub-cent gas) and the yield is real sDAI — the honest end-to-end test.
Phase 0 · ship the code

Push, then let CI place the deployer

The wizard needs the overrides-capable CrowdStakeDeployer on-chain. That's fully pipeline-managed — no manual address editing.

  • Nothing to merge — this work ships in this repo (the BreadchainCoop ourcoop fork). A push to main touching src/** auto-triggers deploy-frontend.yml → the wizard UI ships to GitHub Pages.
  • Run the contracts workflow. Actions → Deploy contracts (multi-chain) → target DeployCrowdStakeDeployer, chain gnosis.
  • Its publish step writes the new deployer address into the rolling contract-addresses release.
  • remote-addresses.ts hydrates CHAINS[100].deployer from that release at runtime — the live wizard picks it up on the next page load. No rebuild for the address swap.
You can also deploy it locally with the same script if you'd rather not wait for CI:
# from contracts/
PRIVATE_KEY=$PK forge script script/DeployCrowdStakeDeployer.s.sol \
  --rpc-url https://rpc.gnosischain.com --broadcast --slow -vvvv

Phase 1 · sanity check

Confirm the modules are ready

Reads only — no gas. Confirms the two “to-be-initialized” modules really are blank, and the token points at real WXDAI/sDAI.

# token: COVA USD / cUSD, over WXDAI→sDAI, claimer not yet set
cast call $TOK 'name()(string)'   --rpc-url $RPC
cast call $TOK 'symbol()(string)' --rpc-url $RPC
cast call $TOK 'yieldClaimer()(address)' --rpc-url $RPC   # → 0x000…000

# voting module + strategy are uninitialized
cast call $VM    'getVotingPowerStrategies()(address[])' --rpc-url $RPC  # → []
cast call $STRAT 'owner()(address)' --rpc-url $RPC                 # → 0x000…000

# registry initialized, empty
cast call $REG 'getRecipientCount()(uint256)' --rpc-url $RPC  # → 0
If those match (empty strategies, zero claimer/owner, count 0), you're clear to run the wizard.
Phase 2 · the wizard

Open “Deploy your instance”

On the live app (once the frontend has shipped), connect your wallet as the coordinator on Gnosis and go to the deploy wizard — from these docs that's ../app/deploy/ (the app's /app/deploy route).

  • Set a short Cycle length for testing — e.g. 2 minutes (~24 blocks at 5s/block). You can bump it later per instance.
  • Leave Owner as you (the coordinator).
  • Expand “Custom modules (advanced).”
As soon as you paste an override, the superseded canonical inputs disappear — the registry override hides the governance picker, the token override hides name/symbol, and the strategy override locks yield distribution to “By votes.” That's expected.

Phase 2 · capture the output

Grab the distribution-manager address

On the success screen, open “Contract addresses” and copy the Distribution Manager. Everything downstream hangs off it — it's also the instance's shareable id (?i=…&c=100).

# add it to your shell env
export DM=0x…distribution-manager-from-the-success-screen…
The success screen also shows a “Finish wiring your custom modules” checklist — that's your next three actions. The Authorize yield claimer button there does follow-up #1 for you; the other two are quick cast calls.
Phase 3 · finish wiring

Three follow-ups

The deployer can't touch your caller-owned modules or initialize contracts that needed the DM address — so you close the loop. Run as the coordinator.

# 1 · let the manager claim yield from the token
#     (or just click "Authorize yield claimer" on the success screen)
cast send $TOK 'setYieldClaimer(address)' $DM --rpc-url $RPC --private-key $PK

# 2 · initialize the Art-Fund strategy (topN = 5)
cast send $STRAT 'initialize(address,address,address,uint256)' \
  $TOK $DM $COORD 5 --rpc-url $RPC --private-key $PK

# 3 · initialize the voting module with the membership strategy
cast send $VM 'initialize(address[],address,address)' \
  "[$POWER]" $DM $COORD --rpc-url $RPC --private-key $PK
The canonical cycle module is already wired by the deployer — nothing to do there.
Phase 3 · verify the wiring

Confirm it's a live, coherent system

This is exactly the read-set the app uses to resolve an instance. All should now point at your modules.

cast call $DM 'baseToken()(address)'          --rpc-url $RPC  # → $TOK
cast call $DM 'votingModule()(address)'       --rpc-url $RPC  # → $VM
cast call $DM 'recipientRegistry()(address)'  --rpc-url $RPC  # → $REG
cast call $DM 'distributionStrategy()(address)' --rpc-url $RPC  # → $STRAT
cast call $TOK 'yieldClaimer()(address)'       --rpc-url $RPC  # → $DM
cast call $VM  'getVotingPowerStrategies()(address[])' --rpc-url $RPC  # → [$POWER]
Phase 4 · seed the cooperative

Add members (one person, one vote)

Members are the addresses allowed to vote. The coordinator owns the membership contract.

# pass every member address in one array (include yourself to vote)
cast send $POWER 'addMembers(address[])' \
  "[$COORD,0xMEMBER_TWO,0xMEMBER_THREE]" \
  --rpc-url $RPC --private-key $PK

# check
cast call $POWER 'getCurrentVotingPower(address)(uint256)' $COORD --rpc-url $RPC # → 1000000000000000000 (1e18)
Each member returns exactly 1e18 voting power — equal weight, regardless of holdings.
Phase 4 · seed the cooperative

Register the art projects

registerProject(recipient, fullBudget, minViableBudget, title, summary) — then processQueue() to activate them at the cycle boundary. Budgets in cUSD (1e18 = 1).

cast send $REG 'registerProject(address,uint256,uint256,string,string)' \
  0xPROJECT_1 5000000000000000000000 2000000000000000000000 "Mural" "Textile-mill mural" \
  --rpc-url $RPC --private-key $PK

# …repeat for each project…

# activate the queued projects
cast send $REG 'processQueue()' --rpc-url $RPC --private-key $PK
cast call $REG 'getRecipientCount()(uint256)' --rpc-url $RPC   # → number of projects
Min-viable budget is the floor: if a project's vote-share would fund it below this, it's dropped and its share is redistributed to the projects that clear their floor. That's the O.U.R.COOP allocation rule.
Phase 5 · put money in

Stake principal & mint cUSD

Wrap xDAI → WXDAI, approve the token, mint cUSD. The principal is parked in sDAI; the interest it earns is what gets distributed.

# amount = 100 cUSD here (1e20). Scale as you like.
export AMT=100000000000000000000

cast send $WXDAI 'deposit()' --value $AMT --rpc-url $RPC --private-key $PK       # xDAI → WXDAI
cast send $WXDAI 'approve(address,uint256)' $TOK $AMT --rpc-url $RPC --private-key $PK
cast send $TOK   'mint(address,uint256)'    $COORD $AMT --rpc-url $RPC --private-key $PK

cast call $TOK 'balanceOf(address)(uint256)' $COORD --rpc-url $RPC   # → your cUSD
Real yield is slow. On mainnet, distributable yield = sDAI growth above principal, which accrues over time. For a quick functional test, distribution only needs yield ≥ (number of projects) wei — so even a short wait works; the payouts will just be small. For big, exact numbers, stake more or wait longer.
Phase 5 · govern

Cast votes

Each member allocates a 100-point ballot across the projects (index-aligned with the registry order). Two entrypoints — both work:

# canonical entrypoint (what the app's Vote page calls)
cast send $VM 'voteWithData(uint256[],bytes)' "[20,15,13,12,9,8]" 0x \
  --rpc-url $RPC --private-key $MEMBER_KEY

# or the COVA module's direct entrypoint (supports re-casting within a cycle)
cast send $VM 'castVote(uint256[])' "[18,14,13,12,9,7]" \
  --rpc-url $RPC --private-key $MEMBER_KEY
The ballot total must be ≤ 100. Unlike the canonical module (per-recipient cap only), the cooperative ballot's sum is capped at 100 — the point array above sums to 77. Over 100 reverts.
Phase 5 · pay out

Advance the cycle & distribute

Once the cycle length has elapsed (blocks) and there's at least a wei of yield per project, anyone can trigger the round.

# ready only when: cycle complete + votes cast this cycle + yield ≥ #projects
cast call $DM 'isDistributionReady()(bool)' --rpc-url $RPC   # → true

cast send $DM 'claimAndDistribute()' --rpc-url $RPC --private-key $PK

# check a project got funded (cUSD minted straight to the recipient)
cast call $TOK 'balanceOf(address)(uint256)' 0xPROJECT_1 --rpc-url $RPC
cast call $DM  'cycleManager()(address)' --rpc-url $RPC   # cycle module; getCurrentCycle() bumps to 2
This is the exact path the fork test drove against these live contracts — it reproduced the worked example (Mural 2584, Theatre 1972, Zine capped at 1200, Sculpture dropped below its floor) to the wei against real sDAI.
Phase 5 · use it in the app

Drive it from the UI

The instance is a standalone, shareable page. Open the link the wizard gave you (or build it):

https://<your-app>/app/?i=$DM&c=100
# or, relative to these docs:
../app/?i=$DM&c=100
  • Deposit — wrap & mint cUSD from the UI instead of cast.
  • Vote — the 100-point ballot; the page reads maxPoints() and calls voteWithData.
  • Recipients — see the registered projects.
  • Distribute — trigger the round when it's ready.
The app resolves the whole instance from just $DM (via resolveInstance) and stores it in your instance switcher — the same flow the wizard used to register it on the success screen.
Recap · one-screen cheat sheet

The whole flow

Ship

  • Push to main (this repo) → frontend auto-deploys
  • Run contracts-deploy (DeployCrowdStakeDeployer) → deployer indexed in the release manifest

Wizard

  • ../app/deploy/ → Custom modules → paste 5 addresses → Deploy
  • Save $DM from the success screen

Wire (as coordinator)

  • TOK.setYieldClaimer($DM)
  • STRAT.initialize(TOK,$DM,COORD,5)
  • VM.initialize([POWER],$DM,COORD)

Seed & run

  • POWER.addMembers([...])
  • REG.registerProject(...) ×N → processQueue()
  • mint cUSD → vote → wait → DM.claimAndDistribute()
If distribution won't trigger: check isDistributionReady() — you need (1) the cycle elapsed, (2) at least one member vote this cycle, and (3) yield ≥ number of projects. All three, or it reverts DistributionNotReady.
